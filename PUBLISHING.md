# Публикация Bloq в Chrome Web Store

Пошаговая инструкция. Часть шагов может сделать только владелец аккаунта (оплата,
загрузка, заполнение форм) - я подготовил всё, что можно подготовить заранее.

## 0. Что уже готово в репозитории

- `npm run build` -> `dist/bloq-<версия>.zip` - **готовый пакет для загрузки** (только
  рантайм-файлы, без тестов и dev-мусора, manifest.json в корне архива).
- `store-assets/` - скриншоты 1280x800 и малый промо-тайл 440x280 (PNG).
- `icons/icon128.png` - иконка магазина 128x128.
- `STORE_LISTING.md` - готовые тексты: название, краткое и полное описание (RU + EN),
  категория, single purpose, обоснования разрешений, заявление о данных.
- `PRIVACY.md` - политика конфиденциальности (её URL нужен магазину; см. шаг 4).

## 1. Аккаунт разработчика (разово)

1. Зайди на <https://chrome.google.com/webstore/devconsole> под своим Google-аккаунтом.
2. Прими соглашение разработчика и **оплати разовый регистрационный взнос $5**
   (без него публиковать нельзя).
3. Заполни профиль разработчика (имя/email для связи) - Google требует верифицированный
   контактный email.

## 2. Загрузка пакета

1. Собери свежий пакет: `npm run build`.
2. В Developer Dashboard -> **Add new item** -> перетащи `dist/bloq-<версия>.zip`.
3. Дождись, пока пройдёт первичная проверка манифеста.

## 3. Заполнение карточки (Store listing)

Бери готовый текст из `STORE_LISTING.md`:

- **Name**: Bloq
- **Summary** (краткое, <=132 символа): поле «summaryRu».
- **Description**: поле «descriptionRu» (для англоязычной витрины - «descriptionEn»).
- **Category**: Privacy & Security.
- **Language**: Russian (можно добавить English как дополнительный перевод).
- **Icon**: `icons/icon128.png`.
- **Screenshots**: залей PNG из `store-assets/` (нужен минимум 1, размер 1280x800 или
  640x400). Малый промо-тайл 440x280 - опционально.

## 4. Privacy / вкладка Practices

- **Single purpose**: поле «singlePurpose» из `STORE_LISTING.md`.
- **Permission justifications**: для каждого разрешения (declarativeNetRequest, storage,
  activeTab) вставь обоснование из «permissionJustifications».
- **Data usage**: отметь, что расширение **не собирает пользовательские данные**
  (формулировка - в поле «dataUsage»); проставь галочки соответствия политике.
- **Privacy policy URL**: обязателен. После пуша репозитория используй прямую ссылку на
  `PRIVACY.md`, например:
  `https://github.com/<user>/<repo>/blob/main/PRIVACY.md`
  (страница должна быть публично доступна - значит репозиторий должен быть публичным,
  либо размести политику на любом публичном URL).

## 5. Отправка на ревью

1. Проверь превью карточки.
2. Выбери видимость (Public / Unlisted) и регионы.
3. **Submit for review**. Проверка обычно занимает от нескольких часов до нескольких дней.
   Расширения с declarativeNetRequest и правами обычно проходят автоматизированную +
   иногда ручную проверку.

## 6. Обновления в будущем

1. Подними `version` в `manifest.json` (и в `package.json`, `CHANGELOG.md` - их сверяет
   тест `npm test`).
2. `npm run build` -> загрузи новый zip в том же item -> Submit.

## Частые причины отказа (и почему у нас их быть не должно)

- **Избыточные разрешения** - у нас только 3 узких разрешения, без широких host-прав.
- **Непрозрачный сбор данных** - мы ничего не собираем, это заявлено.
- **Несоответствие single purpose** - расширение делает ровно одно: блокирует
  рекламу/баннеры/cookie-окна.
- **Битый пакет** - `npm test` проверяет целостность манифеста и наличие всех файлов
  перед сборкой.
